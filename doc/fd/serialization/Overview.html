Overview

FD.Serialization Library

The latest CHM version of this documentation is available here.

FD.Serialization Assembly, together with its satellite assemblies, is originally created to provide some kind of inversion of control mechanism when it comes to serialization and serialization products available. We didn't wanted our code to be tightly dependant on Google Protocol Buffers for example; instead we wanted to be able to change actual serialization mechanism at any point in time without having to change single line of code in our software that uses serialization. Basically, the original idea was to create a library that will define interfaces for usual serialization operations, and then let the library to choose actual serialization mechanism that will be used. But as soon as we've started with development it turned out that the library will introduce few more features, and solve few more problems. Finally we've ended up with library that is true life-savior in developing complex, heavily loaded, still performant distributed software systems. FD.Serialization library has became our flagship product among libraries from our development framework.

Core Library and Satellite Libraries

Before explaining the components included I need to emphasize one thing: FD.Serialization library and its satellite assemblies do not introduce any new serialization mechanism. Instead they implement existing serialization products internally (for example, FD.Serialization.Protobuf Assembly is based on existing Google Protocol Buffers library). Let's see how everything is organized:

  • FD.Serialization Assembly is core library that provides a framework to use for developing satellite assemblies. Also it implements some features that can be used no matter which satellite library is used (i.e. compression, etc.)
  • Serializer assemblies (or satellite assemblies) are assemblies which are implementing particular existing serialization product. For example, FD.Serialization.Protobuf Assembly is a serializer assembly that implements Google Protocol Buffers library. Serializer assemblies basically represent a bridge between particular existing serialization library (Google Protobuf in our example), and core FD.Serialization library.

Audience

Before jumping into details about the library I need to define three possible groups among readers of this manual:

  • IT Enlight employees are those among you who are lucky to work for IT Enlight, and who will continue with developing the base library and its satellites which are under our control;
  • Serializer library developers are those of you who want to develop another serializer assembly (probably one that implements some other serialization mechanism);
  • End users are those who'll simply use FD.Serialization library and existing satellite assemblies in their projects.

Obviously in all three groups we are talking about developers. It is important to remember these terms because they'll be used throughout this manual. For example if you want to learn how to use this library, and not to develop new serializer assemblies, and you encounter a section throughout this documentation that is marked as content for serializer library developers - you can skip this section entirely, still being sure that you haven't skipped something that you should know.

Content Structure

There is a lot to say about the library and its features, and I'll start with the most important thing:

DON'T PANIC

Everything will be represented in comprehensive way, and you can master everything (in theory) in one day. Practice will do the rest.

There are quite a few questions we need to cover (answer) here:

  • Why to use the library (what valuable it has to offer)?
  • When to use the library?
  • How to implement it?
  • How to configure it?
  • How it affects performances?

For this manual I've decided not to go with explaining one question completely before proceeding to another. Instead I've decided to go in feature-by-feature manner. It means that for every major feature I will:

  • Describe the feature (what it is useful for);
  • Provide some comparative view how things would be done without the feature, and with the feature, so that you'll immediately know why it is nice to have and use the feature (or to skip the rest if you are not interested in);
  • Describe how the feature affects performance and provide benchmark tests where appropriate;
  • Enumerate and explain configuration options related to this particular feature. It means that there won't be a dedicated section that explains configuring FD.Serialization library; instead every section that covers some of the features will have part that deals with configuring this feature.

While explaining features of the library I'll often compare it with .NET embedded serialization mechanism (i.e. BinaryFormatter), and Google Protocol Buffers. I've intentionally chosen these for the following reasons:

  • .NET embedded mechanism is by far the most convenient one to implement and use. Basically the only thing that needs to be done is to mark a class with Serializable attribute, and .NET framework will do everything internally and completely transparently. But on the other side this mechanism has few huge drawbacks, the most important being:
    • Performances - it's way too slow and too CPU intensive for any merely demanding project - it's often hundreds or even thousand times slower than Google Protocol Buffers;
    • Message size - byte array (or stream) gotten is too big - it's often dozen times larger than the result of Google Protocol Buffers serialization;
    • Obsolete - Microsoft has discontinued binary serialization in .NET Core, and they are suggesting using other serialization mechanisms including Google Protocol Buffers (as you can read in "Binary Serialization" section in this blog).
  • Google Protocol Buffers is probably the most popular serialization mechanism these days, and very powerful when it comes to performances, message size, etc. But it's not so simple to implement it, and even worse - it's not so simple to use it (as will be shown in this manual).
