Remarks
This widget is used in request-reply communication (at reply side), but it implements ROUTER socket instead of REP socket. Such setup is described in The REQ to ROUTER Combination section of ZeroMQ guide. The reason for using ROUTER socket instead of REP socket is that ROUTER socket allows accepting multiple requests simultaneously. The key difference comparing to ReplyWidget is in the way these widgets operate internally:
  • ReplyWidget internally uses REP socket, thus inheriting its limitation: messages can flow only in receive-send-receive-send order. It means that if we simultaneously send several messages, ReplyWidget will process them as follows:
    • It will receive the first message and call the handler function;
    • Than it will wait while the handler function finishes and returns the response;
    • It will then send the response, and receive the second message;
    • etc.
  • RouterAsRepWidget internally implements ROUTER socket which allows it to receive more than one message without waiting for handler function to finish the previous response. So if it simultaneously receive several messages RouterAsRepWidget will immediately call the handler function in multiple threads, and than it will wait for responses.
The difference is obvious - RouterAsRepWidget runs the handler function in multiple threads while ReplyWidget runs it in single-thread fashion, waiting for the result of the previous call before making the next call. If the handler function is thread-safe and supports multi-threading - RouterAsRepWidget will offer significantly better performances than ReplyWidget. This widget fakes REP socket so that destination endpoint can't see any difference. It means that you can use this widget wherever REP socket can be used. Be aware that with RouterAsRepWidget the handler function has to be multi-threading capable. RouterAsRepWidget will immediately call the handler function as soon as new request arrives, without any control, meaning that if it receives huge number of messages fast - it'll initiate a huge number of the handler function calls (huge number of threads).
