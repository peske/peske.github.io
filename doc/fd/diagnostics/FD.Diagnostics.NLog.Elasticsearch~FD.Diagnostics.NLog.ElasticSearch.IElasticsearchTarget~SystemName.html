Remarks
FD.Diagnostics.NLog.ElasticSearch assembly automatically creates all necessary ElasticSearch indices and document types, and while doing this it will use the SystemName for naming the indices and aliases. Although it is not a requirement, it is strongly recommended to use short system names (up to 10 chars), letters and numbers, all lowercase. Assuming that SystemName is set to "myapp", ElasticSearch index name and aliases will be created in the following way:
  • Depending which target is used (ElasticsearchTarget or ElasticsearchTargetEx), the actual index name and assigned aliases will be prefixed with fdlog- or fdlogex-, respectively.
  • If SystemName is not null or empty, it will be appended: fdlog-myapp- or fdlogex-myapp-.
  • The actual index name will be created by appending current date: fdlog-myapp-2016.04.19 or fdlogex-myapp-2016.04.19. Document type defined by the index will be named fdlogEvent or fdlogexEvent.
  • The following aliases will be assigned to the index:
    • fdlog-myapp-today
    • fdlog-myapp-week
    • fdlog-myapp-all
    • fdlog-today
    • fdlog-week
    • fdlog-all
    Of course, if ElasticsearchTargetEx is used, the aliases would be prefixed with fdlogex- instead of fdlog-.
The final thing to clarify regarding this property is its type (Layout), although System.String looks as a better choice, right? Well, reason for such implementation is to allow dynamic resolution of the application's system name at runtime. For example, you may decide to have different ElasticSearch index for different loggers in your application. This can be accomplished by setting SystemName property to something like "myapp-${logger}". (I doubt that this can be a good idea, but still it is possible.) For complete list of layout renderers that can be used see Layout Renderers section of NLog wiki site. Of course, if such runtime name resolution is not needed you can simply use System.String value. Unless you are sure that you know what are you doing use simple System.String application names.
