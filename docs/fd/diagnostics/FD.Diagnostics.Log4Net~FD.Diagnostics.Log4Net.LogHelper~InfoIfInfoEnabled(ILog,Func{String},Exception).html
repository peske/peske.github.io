Remarks
In some cases the message to be logged is not simple as string literal, but instead it is generated in some way that can influence performances. For example, if we need to loop through some collection(s) in order to generate the message to be logged. On the other hand, after the message is created it may happen that it will not be logged at all because of its logging level, so all the effort invested in creating the message is wasted for nothing. For this reason it is good practice to check if the message will be logged before creating the message itself, as in the following example: if (logger.IsDebugEnabled) { string message = null; // Do any message-generating logic logger.Debug(message); } Obviously, instead of having single line of code that performs logging, now we have few lines. In such situations one of the extension methods exposed by this class can save few lines of code, as in the following example: logger.DebugIfDebugEnabled(() => { // code that generates and returns the message }); Both previous code examples have the same effect - the massage-generating code will be executed only if it is confirmed that the logging will happen.

The examples above are for log4net.Core.Level.Debug level, but the same stands for all other levels.
